## Bisection Methodology

Earlier builds boot-looped or had partial functionality. To isolate variables, we ran a 5-stage bisection:

| Stage | Kernel base | Toolchain | Inject patches | CFI/LTO | Result |
|-------|-------------|-----------|----------------|---------|--------|
| 0 | (untouched LOS slot_a) | — | — | — | Baseline ✓ |
| 1 | (untouched LOS slot_b) | — | — | — | Slot_b super empty ❌ — OTA writes inactive slot only; abandoned slot_b boot tests |
| 2 | LOS upstream (kernel rebuilt as-is) | AOSP-Clang r536225 | none | on | ✅ Boots identically to LOS' own kernel — proves build pipeline correctness |
| 3 | kimocoder fork `nethunter-23.0` (THIS) | AOSP-Clang r536225 | applied | on | ❌ Bootloop — fork's lineage-23.0 base too old; missing `BPF_JIT_DEFAULT_ON`, `BPF_LSM`, `TASKS_TRACE_RCU` |
| 4 | LOS upstream + Kali inject 17-patch series | AOSP-Clang r536225 | applied | on | ✅ Boots, sensors fine, networking fine — **PERSISTED** to slot_a |
| 4-NoCFI | same source as Stage 4, CFI=off | AOSP-Clang r536225 | applied | off | ✅ Boots, but **sensors/battery/nav-bar icons broken** (LOS vendor modules expect CFI=on) |
| 5 | Stage 4 + Realtek modules | — | — | on | ✅ Modules load; ❌ `iw set type monitor` on wlan1 → **kernel panic + reboot** (CFI signature mismatch on indirect call) |

**Conclusion:** Stage 4 is the kernel; Stage 5 panic was the final blocker.

## Bug & Fix Catalogue

### Bug 1 — `make distclean` wipes Focaltech firmware blob

The Focaltech touchscreen driver `drivers/input/touchscreen/focaltech_touch/focaltech_flash.c` `#include`s a 598 KB header `FT3680_WXN_M146_V27_D01_20220706_app.i`. The header is **missing** from the `kimocoder/android_kernel_lineage_nothing_sm7325` source tree. `make distclean` wipes it across rebuilds.

**Fix** (in `scripts/03-apply-patches.sh`): re-fetch from `kimocoder/android_kernel_msm-5.4_nothing_sm7325` (where it IS committed):

```bash
curl -sLf "https://raw.githubusercontent.com/kimocoder/android_kernel_msm-5.4_nothing_sm7325/nethunter-15.0/drivers/input/touchscreen/focaltech_touch/include/firmware/FT3680_WXN_M146_V27_D01_20220706_app.i" -o $FW_DIR/FT3680_WXN_M146_V27_D01_20220706_app.i
touch $FW_DIR/fw_sample.i
```

### Bug 2 — `haven/hh_msgq.h` stub functions return wrong type

5 stub functions declared `static inline int` but `return ERR_PTR(...)` (which returns `void *`). Compile error.

**Fix:** sed-replace `ERR_PTR(-ENODEV)` → `-ENODEV`, etc., for `hh_msgq_unregister/send/recv/populate_cap_info/probe`.

### Bug 3 — `msm_cvp_ioctl.c` missing `<linux/compat.h>`

Uses `compat_ptr()` without the include.

**Fix:** sed-insert `#include <linux/compat.h>` at line 6.

### Bug 4 — Realtek Makefiles use GCC-only flags

- `-Wno-stringop-overread` — older Clang doesn't recognize
- `-Wno-enum-int-mismatch` — GCC-only; AOSP Clang errors out as `-Werror=unknown-warning-option`
- `-DRTW_ENABLE_WIFI_CONTROL_FUNC` — forces inclusion of `wlan_plat.h` (Android-internal) we don't have
- rtl8188eus has no ARM64 platform block

**Fix:** `scripts/03-apply-patches.sh` sed-strips these and injects an ARM64 platform block at the top of the rtl8188eus Makefile.

### Bug 5 — `qcacld-3.0` Kbuild rejects Kali patches' enum types under `-Werror`

Kali inject patches use older enum types that newer LOS qcacld-3.0 doesn't allow under `-Werror=enum-conversion`.

**Fix:** prepend `ccflags-y += -Wno-enum-conversion` to `drivers/staging/qcacld-3.0/Kbuild`.

### Bug 6 — Kali QCACLD inject patch 02/17 already cherry-picked into lineage-23.0

`git am --3way --keep-cr` fails on patch 02 because it's already applied.

**Fix:** loop `git am --skip` up to 20 times; if still stuck, fall back to `patch -p1 --forward --no-backup-if-mismatch`. All 17 patches end up in HEAD even with skips.

### Bug 7 — MSYS path conversion in Git Bash

Git Bash on Windows mangles `/work` → `C:/Program Files/Git/work` inside `docker exec` heredocs.

**Fix:** every script and every `docker exec` heredoc starts with `export MSYS_NO_PATHCONV=1`.

### Bug 8 — `docker exec` without `-i` swallows heredoc stdin

Without the `-i` flag, the heredoc isn't piped to the container shell; container exits without doing anything.

**Fix:** all `docker exec` calls use `docker exec -i`.

### Bug 9 — sed `\n` ate the `n` before `export`

Produced literal `nexport` instead of newline + `export`.

**Fix:** explicit `sed -i 's/^nexport/export/g'` cleanup pass.

### Bug 10 — kimocoder fork (`nethunter-23.0` of `android_kernel_lineage_nothing_sm7325`) is too old

Boot-loops on Stage 3 because its lineage-23.0 base predates the BPF JIT and TASKS_TRACE_RCU work LOS 23.2 vendor modules need.

**Fix:** abandon the fork. Use upstream LineageOS source + apply Kali QCACLD inject patches via `git am`.

### Bug 11 — Stage 4-NoCFI breaks LOS vendor modules

Sensors, battery icon, network/signal indicators in nav-bar all dead. Confirmed by user: *"Сенсор в отвале, значки батареи сети и всё в отвали"*.

**Cause:** LineageOS 23.2 vendor modules (icnss2, audio HAL, etc.) are linked with CFI and refuse to load on a CFI=off kernel.

**Fix:** keep CFI=on. Stage 4-NoCFI cannot be used as the daily kernel.

### Bug 12 — CFI panic on `iw set type monitor` (Realtek wlan1)

Kernel panic + reboot when calling `iw dev wlan1 set type monitor` on Realtek-driven interface. Confirmed by user: *"Проверяй монитор мод"* → followed by reboot.

**Cause:** Realtek out-of-tree drivers (rtl8188eus, 88x2bu, 8821cu) declare callback functions with **void * tasklet** signature and **`int` return / non-skb args** for `ndo_start_xmit`. Kernel's CFI layer expects:
- `tasklet_callback(unsigned long)`
- `netdev_tx_t(*)(struct sk_buff *, struct net_device *)`

Indirect call through tasklet/netdev_ops checks the signature → mismatch → panic.

**Fix:** apply the **CFI signature fix** from `aircrack-ng/rtl8812au` PR #1041 (Author: GeorgeBannister) to all 3 drivers via sed/awk. Six changes per driver:

```c
/* include/usb_ops_linux.h */
-void usb_recv_tasklet(void *priv);
+void usb_recv_tasklet(unsigned long priv);

/* os_dep/linux/usb_ops_linux.c (2 occurrences) */
-void usb_recv_tasklet(void *priv)
+void usb_recv_tasklet(unsigned long priv)

/* os_dep/linux/xmit_linux.c */
-int _rtw_xmit_entry(_pkt *pkt, _nic_hdl pnetdev)
+netdev_tx_t _rtw_xmit_entry(struct sk_buff *pkt, struct net_device *pnetdev)
-int rtw_xmit_entry(_pkt *pkt, _nic_hdl pnetdev)
+netdev_tx_t rtw_xmit_entry(struct sk_buff *pkt, struct net_device *pnetdev)
/* and inside rtw_xmit_entry: */
-	int ret = 0;
+	netdev_tx_t ret = NETDEV_TX_OK;

/* include/xmit_osdep.h */
-extern int _rtw_xmit_entry(_pkt *pkt, _nic_hdl pnetdev);
-extern int rtw_xmit_entry(_pkt *pkt, _nic_hdl pnetdev);
+extern netdev_tx_t _rtw_xmit_entry(struct sk_buff *pkt, struct net_device *pnetdev);
+extern netdev_tx_t rtw_xmit_entry(struct sk_buff *pkt, struct net_device *pnetdev);
```

The full diffs are saved as patches in `artifacts/realtek-patches/`. After applying + rebuilding, monitor mode + tcpdump capture worked **without panic**.

### Bug 13 — Module CRC mismatch ("disagrees about version of symbol module_layout")

Modules built today refused to load on the running kernel even though vermagic strings matched.

**Cause:** the build out-tree's `Module.symvers` was from a *later* kernel rebuild (Stage 4-NoCFI) than the boot.img on slot_a (original Stage 4 CFI=on). Different build → different `module_layout` CRC.

**Fix (deterministic-match build):**

1. On the phone, with the working kernel running:
   ```bash
   adb shell "su -c 'cat /proc/config.gz' > running-config.gz"
   adb pull /sdcard/running-config.gz
   ```
2. In container, replace `/work/kernel-los/out/.config` with the gunzipped running config.
3. `make olddefconfig` (no changes — config is from a self-consistent build).
4. `make Image modules` — this rebuilds the kernel **with the same config that produced the running kernel**, so `Module.symvers` regenerates with **byte-identical CRCs**.
5. Build Realtek modules against this `Module.symvers` — module_layout CRC matches → modules load **without rebuilding boot.img**.

Critical detail: the `-g192e5b024436-dirty` suffix means HEAD is `192e5b024436` AND the git tree must be dirty. We `touch Makefile` to force the dirty flag; otherwise the build produces `5.4.302-qgki-g192e5b024436` (no `-dirty`) and vermagic mismatches.

The full running config lives at `running-config.gz` in the repo root.

### Bug 14 — Repacked boot.img with new Image froze on splash

We rebuilt the kernel with a config drift (used `vendor/lahaina-qgki_defconfig` + `debugfs.config` + scripts/config overrides + `LOCALVERSION=-NetHunter`), produced a new `Image`, repacked the original Stage 4 boot.img with magiskboot, RAM-booted via `fastboot boot`. Result: splash screen froze indefinitely.

**Cause:** even with CFI=on and LTO=on enabled, the resulting `Image` had different code layout/symbols than the original Stage 4 because:
- `LOCALVERSION="-NetHunter"` → vermagic `5.4.302-NetHunter-NetHunter` (double — `LOCALVERSION` env var stacks with `--set-str LOCALVERSION` in scripts/config)
- Subtle `vendor/lahaina-qgki_defconfig + debugfs.config + olddefconfig` differences vs original

**Fix:** abandon the from-scratch reconfigure approach. **Always use `running-config.gz` from the running kernel** as the deterministic source of truth. The new build will be byte-equivalent to what's on slot_a.

### Bug 15 — Internal WiFi monitor mode requires `con_mode=4` module reload

Standard `iw dev wlan0 set type monitor` returns `Operation not supported on transport endpoint (-95)` even though `iw phy phy0 info` lists `monitor` in supported modes. This is **by design** in QCACLD-3.0 — see Kali patch 16/17 *"wlan: Asynchronous synchronization of Wi-Fi mode (STA/MONITOR), bypassing iw commands"*.

**Workflow (works):**
```bash
svc wifi disable
ip link set wlan0 down
rmmod wlan
insmod /data/local/tmp/wlan.ko con_mode=4   # 4 = MONITOR (was 0 = STA)
svc wifi enable     # triggers cnss-daemon to start firmware → wlan0 reappears in monitor mode
iw dev wlan0 info | grep type     # → type monitor
ip link set wlan0 up
iw dev wlan0 set channel 6
tcpdump -i wlan0 -c 8 -nn         # ✅ captures real WPA2 frames in air
```

The freshly-built `wlan.ko` (15 MB stripped, vermagic `5.4.302-qgki-g192e5b024436-dirty`) is in `output/wlan.ko`. con_mode values: `0=MISSION (STA)`, `4=MONITOR`, `5=FTM`, `6=EPPING`.

### Bug 16 — Internal WiFi inject (TX) panics the kernel

`aireplay-ng --test wlan0` (or any inject through standard nl80211 `pcap_inject`) → kernel panic → phone reboot. Confirmed by user: *"Опа упало, телефон перезагрузился"*.

**Cause:** same class of CFI signature mismatch as Bug 12, but in the QCACLD-3.0 inject codepath rather than Realtek. Standard userspace tools (`aireplay-ng`, `hostapd`, etc.) call into the inject path through generic mac80211 wrappers; the function pointer signatures in QCACLD don't match what CFI expects → panic on first indirect call.

**Why it worked on the previous kimocoder XOS-14.0.2 setup:** that kernel was built with Neutron-Clang-19, which has CFI=off by default. With no CFI checks, mismatched signatures don't panic. But **CFI=off is NOT acceptable on LOS 23.2** because vendor modules require it (Bug 11).

**Status:** UNRESOLVED. Two paths forward:
1. **CFI fix patches for QCACLD inject codepath** — find every function pointer with mismatched signature in the inject path, write source patches like PR #1041 did for Realtek. Multi-hour reverse-engineering task.
2. **Use NetHunter app's built-in wireless attack UI** — those ride through Kali's custom `hdd_frame_inject_netlink` API which the patch authors wrote with correct CFI signatures. (Not field-tested today.)

**Practical workaround:** WN722N (Realtek) handles all inject scenarios — it's the proven-working path right now.

| Use case | Internal `wlan0` | External WN722N |
|----------|------------------|-----------------|
| Scan / monitor | ✅ | ✅ |
| Capture handshake | ✅ | ✅ |
| Deauth inject | ❌ panic | ✅ |
| Evil twin (`hostapd-mana`) | ❌ panic | ✅ |
| WPS bruteforce (`reaver`/`bully`) | ❌ panic | ✅ |

### Bug 17 — Magisk module install ZIP didn't fully unpack

`magisk --install-module realtek-wifi-cfi-fix-v1.0.zip` extracted `module.prop` and `update` marker but skipped `system/` and `post-fs-data.sh`.

**Fix:** manually `unzip -o /sdcard/Download/realtek-wifi-cfi-fix-v1.0.zip` inside `/data/adb/modules/realtek-wifi-cfi-fix/` after install. After reboot, post-fs-data.sh runs and the modules `insmod` from `system/lib/modules/`. Verified working: 8188eu and 8821cu auto-load on every boot.

(Possibly the ZIP needs different META-INF structure — but the manual-unzip workaround is reliable.)

### Bug 18 — WiFi adb port rotates each session on Android 16

Wireless debugging exposes a different random port every time it's toggled. Ports observed today: 40413 → 42210 → 42219 → 38539 → (after reboot) something new.

**Workaround:** `adb tcpip 5555` while USB is connected makes a stable port for the session, but wireless-debugging mode resets it. For predictable testing, plug USB.

## Final Working Build Recipe

```bash
# 0. Container is already set up (scripts/01-setup-container.sh)
#    Sources cloned (scripts/02-clone-sources.sh)
# 1. Apply source patches
scripts/03-apply-patches.sh                  # incl. Realtek CFI fix (Bug 12)
                                             # incl. Kali QCACLD inject 17-patch series
                                             # incl. focaltech firmware refetch (Bug 1)

# 2. Use deterministic config from running kernel (Bug 13)
adb shell "su -c 'cat /proc/config.gz' > /sdcard/running-config.gz"
adb pull /sdcard/running-config.gz running-config.gz
docker cp running-config.gz spacewar-build:/tmp/
docker exec -i spacewar-build bash -c '
  cd /work/kernel-los
  gunzip -c /tmp/running-config.gz > out/.config
  unset LOCALVERSION
  ARCH=arm64 make O=out olddefconfig
  touch Makefile  # force -dirty suffix
'

# 3. Build kernel + modules
scripts/05-build-kernel.sh                    # ARCH=arm64 LLVM=1 LLVM_IAS=1, ~13 min on 6 cores

# 4. Build Realtek modules against new Module.symvers
scripts/06-build-modules.sh                   # 3 .ko files in /work/modules-FINAL/

# 5. Push Realtek modules to phone (rebuild boot.img only if kernel itself changed)
adb push output/modules-FINAL/{8188eu,88x2bu,8821cu}.ko /data/local/tmp/

# 6. Or install via Magisk module for auto-load
adb push output/realtek-wifi-cfi-fix-v1.0.zip /sdcard/Download/
adb shell "su -c 'magisk --install-module /sdcard/Download/realtek-wifi-cfi-fix-v1.0.zip'"
# WORKAROUND for Bug 17 — manually unzip into module dir:
adb shell "su -c 'unzip -o /sdcard/Download/realtek-wifi-cfi-fix-v1.0.zip -d /data/adb/modules/realtek-wifi-cfi-fix -x META-INF/*'"
adb reboot
```
