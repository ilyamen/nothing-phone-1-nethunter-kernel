# NetHunter Kernel v1.1.1 — Nothing Phone (1) (spacewar)

Debug visibility release. **Same FULL_MON kernel as v1.1.0** (kernel commit `88ac9aaa6` — internal Wi-Fi monitor RX still works), but:

- 4× larger printk ring (128 KB → 512 KB) — much more dmesg history visible
- Timestamps on every dmesg line (`CONFIG_PRINTK_TIME=y`)
- 4× larger pstore region (1 MB → 4 MB) — full multi-CPU panic dumps survive reboot
- Userspace pmsg ring (512 KB) — `/dev/pmsg0` writes survive panic
- `nh-logcatd` Magisk module bumped to v3 — exploits all new buffers

This release is **purely debug visibility**. Kernel ABI is unchanged from v1.1.0 — vendor modules (sensors, wlan, audio, modem) load identically. No new attack-surface code.

## TL;DR

When a phone bug appears (mobile data flaky, wifi reconnect, sensor weirdness), v1.0.0/v1.1.0 left users with very little to debug from — kernel logs were tiny (128 KB ring), no timestamps, panic forensics fit in 1 MB. v1.1.1 quadruples kernel diagnostic surface without breaking anything.

## Quick Install (upgrading from v1.0.0 / v1.1.0)

Same procedure as before — patch the v1.1.1 boot.img through Magisk and flash. Replace the old `nh-logcatd` module with v3 (Magisk → Modules → install zip).

```bash
adb push spacewar-nethunter-kernel-v1.1.1-boot.img /sdcard/Download/
# Magisk → Install → Patch a File → select v1.1.1 boot.img
adb pull /sdcard/Download/magisk_patched-XXX.img
adb reboot bootloader
fastboot flash boot_a magisk_patched-XXX.img
fastboot reboot

# After phone boots, install nh-logcatd v3:
adb push nh-logcatd-v1.1.1.zip /sdcard/Download/
# Magisk → Modules → Install from storage → select the zip → reboot
```

## What changed

### Kernel build (`scripts/04-configure.sh` — new Batch 9)

```diff
+ CONFIG_LOG_BUF_SHIFT=19         # 128 KB → 512 KB printk ring
+ CONFIG_PRINTK_TIME=y            # dmesg timestamps on every line
+ CONFIG_CMDLINE_EXTEND=y         # extend bootloader CMDLINE
+ CONFIG_CMDLINE="… ramoops.record_size=2097152 ramoops.console_size=524288 \
+                ramoops.ftrace_size=1048576 ramoops.pmsg_size=524288 \
+                panic_print=15"
```

These are the **only safe debug knobs** for this kernel. Every other detector flag (`DETECT_HUNG_TASK`, `SOFTLOCKUP_DETECTOR`, `WQ_WATCHDOG`, `DEBUG_FS`, `DYNAMIC_DEBUG`, etc.) was tested and found to break vendor module CRC (sensors / wlan / audio dead). The reason: prebuilt vendor modules in `/vendor/lib/modules/` are compiled against the running stock LineageOS kernel ABI, and any flag that adds a field to an exported struct (e.g. `DETECT_HUNG_TASK` adds `last_switch_count` to `task_struct`) breaks `MODVERSIONS` matching. Documented for future developers in `scripts/04-configure.sh:255-310`.

### Magisk module: `nh-logcatd` v3

- Logcat ring buffer: 8 MB → **16 MB**
- Persistent logcat pool: 64 MB → **128 MB rotating** (8 files × 16 MB)
- Storage budget: 400 MB → **600 MB**
- New: writes boot marker to `/dev/pmsg0` on every clean boot — survives kernel panic, helps reconstruct timeline
- New: distinguishes `pmsg-only` pstore (= clean reboot marker) from real panic (`dmesg-/console-/ftrace-ramoops`) — no false-positive incidents
- `nh-collect-logs.sh` now snapshots **live `/sys/fs/pstore/`** + `/proc/cmdline`, `/proc/sys/kernel/printk`, wlan params, `/proc/interrupts`, etc.

### What v1.1.1 buys you

| Scenario | v1.0.0 / v1.1.0 | v1.1.1 |
|----------|------------------|--------|
| `dmesg` to see kernel events | Last 128 KB only, no timestamps — when a wifi/modem hiccup happens, by the time you `dmesg` it's already rolled out of the ring | 512 KB ring, every line stamped, ~3-5 minutes of history retained |
| Kernel panic forensics | 1 MB ramoops, only `dmesg-ramoops` captured | 4 MB ramoops: dmesg + console + ftrace + pmsg captured separately |
| Boot timeline reconstruction | Can only see logs from current boot | `nh-logcatd` writes pmsg marker each boot; on next boot we can tell exactly when the last clean boot was, even if mid-boot crash |
| Userspace event correlation | logcat alone | logcat 128 MB rotating + dmesg snapshots every 30s for 24h |

## Hardware tested

- Nothing Phone (1) (A063, codename `spacewar`) — full vendor stack (sensors, wlan, audio, modem) verified post-flash.

## Compatibility

- **Same kernel ABI as v1.1.0** — drop-in upgrade
- LineageOS 23.2 user-builds + Magisk v30+ (same as v1.0.0)
- Other Magisk modules from v1.0.0 (`nh-overlay-base`, `nh-batch5-*`, etc.) — unchanged, no reinstall required
- **Only `nh-logcatd` needs replacement** (v2 → v3)

## Known limitations

- Heavier debug detectors (`DETECT_HUNG_TASK`, `LOCKDEP`, `KASAN`, etc.) cannot be enabled on this kernel without rebuilding the vendor wlan driver (`qca_cld3_qca6750.ko`) and other vendor `.ko`s from source. This is a future v1.x roadmap item if it becomes valuable.
- `DEBUG_FS` and `DYNAMIC_DEBUG` together break vendor stack on this build. Each individually was not separately bisected (the cost was too high relative to the value at this stage). Documented in `scripts/04-configure.sh`.
